## Purpose

An Obsidian plugin that ingests calendar meeting data from two sources — standard ICS/iCal files and
Outlook for Mac plain-text clipboard output — and produces structured Markdown meeting notes
directly in the user's vault. It bridges the gap between external calendar tools and Obsidian's
note-taking workflow, removing the need for manual copy-paste and formatting.

## Domains

This system is organized into the following domains:

| Domain | Description | Spec |
|--------|-------------|------|
| Settings | Renders a drag-and-drop modal UI that lets the user import an .ics file either by dropping it onto the zone or clicking to browse, and delegates the file/event payload to plugin-provided callbacks. | [spec.md](../settings/spec.md) |
| Ical-parser | Parses meeting data from two distinct calendar text formats — iCal/ICS files and Outlook for Mac plain-text clipboard output — into a normalized MeetingEvent structure, and sanitizes meeting descriptions by stripping Microsoft Teams boilerplate content. | [spec.md](../ical-parser/spec.md) |
| Note-creator | Responsible for creating meeting note files in the Obsidian vault: resolving the target folder, rendering a template with event data, injecting structured attendee and invite-notes blocks under configured headings, and writing the resulting Markdown file. | [spec.md](../note-creator/spec.md) |
| I18n | Provides internationalization support by detecting the user's locale from Obsidian/browser context and returning translated strings for supported languages (en, fr, es) with optional variable interpolation. | [spec.md](../i18n/spec.md) |

## Requirements

### Requirement: SystemCapabilities

The system SHALL provide the following capabilities:
- Support two distinct input formats (ICS/iCal via ical.js and Outlook for Mac plain-text) in a single import command, using format detection (presence of 'When:' line) to dispatch to the correct parser
- Strip Microsoft Teams boilerplate from meeting descriptions using locale-aware header matching (en/fr/es), keeping note content clean without requiring user action
- Template-based note generation with a built-in fallback: users can supply a custom vault template file; if absent, a hardcoded default is used — avoids a hard dependency on any template plugin
- Heading-based content injection (attendees block, notes/URL block) keyed on configurable heading strings rather than fixed positions, making notes compatible with arbitrary user templates
- settings.useActiveFolder toggle lets users route new notes to the folder of their currently open file instead of a global target folder, enabling contextual note organisation without extra commands
- I18nService is a shared hub (fanIn=9) called across all domains rather than inlined per domain, centralising locale detection and translation lookup in one place
- Note creation is non-destructive by design: createMeetingNote does not overwrite an existing file with the same name, preventing accidental data loss on duplicate imports
- The dependency graph contains two cycles (noted in static analysis), suggesting some cross-domain coupling that could become a maintenance concern if the plugin grows — uncertain whether these are benign import cycles or meaningful circular dependencies without inspecting the specific nodes

## Technical Notes

- **Architecture Style**: Monolith with domain-separated services. All code runs in a single Obsidian plugin process; there is no client/server split or message bus. The four domains (settings, ical-parser, note-creator, i18n) are organized as cohesive service classes rather than independent modules, which is appropriate given the small surface area and the tight coupling required by the Obsidian plugin API lifecycle.
- **Security Model**: No authentication or authorization layer exists; the plugin operates entirely within the user's local Obsidian vault and never makes outbound network requests for core functionality. Trust is implicit — the plugin runs with the same filesystem permissions as the Obsidian process. The only external data boundary is the raw ICS/text content dragged in by the user; the parser sanitizes Teams boilerplate but performs no further input validation or sandboxing.
- **External Integrations**: Obsidian Plugin API, ical.js, Browser File / DataTransfer API
