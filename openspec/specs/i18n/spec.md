# I18n Specification

> Generated by openlore v1.0.0 on 2026-06-18 09:21

## Purpose

Provides internationalization support by detecting the user's locale from Obsidian/browser context
and returning translated strings for supported languages (en, fr, es) with optional variable
interpolation.
## Requirements

> `src/ts/i18n.ts`

### Requirement: Translate

The system SHALL looks up a translation key in the active locale's strings, falling back to english if the key is absent, and substitutes any provided named variables into the result.

> Implementation: `t` in `src/ts/i18n.ts` · confidence: llm

#### Scenario: KeyFoundInActiveLocale
- **GIVEN** The active locale contains a translation for the given key
- **WHEN** t is called with that key
- **THEN** The locale-specific translation string is returned

#### Scenario: KeyMissingInActiveLocale,FallsBackToEnglish
<!-- openlore-test: tags=regression (auto) -->
- **GIVEN** The active locale does not define a translation for the given key but English does
- **WHEN** t is called with that key
- **THEN** The English fallback string is returned

#### Scenario: VariableInterpolationApplied
- **GIVEN** The translation string contains {{placeholder}} tokens and vars is provided
- **WHEN** t is called with the key and a vars map
- **THEN** Each {{placeholder}} in the string is replaced with the corresponding value from vars

#### Scenario: NoVariablesProvided
- **GIVEN** The translation string may or may not contain placeholder tokens
- **WHEN** t is called with only a key and no vars argument
- **THEN** The raw translation string is returned without any substitution

### Requirement: GetCurrentLocale

The system SHALL returns the locale code that was detected at module initialisation from obsidian's moment instance or the browser's navigator.language, defaulting to 'en' when the detected code is not in the supported locale set.

> Implementation: `locale` in `src/ts/i18n.ts` · confidence: llm

#### Scenario: ObsidianMomentLocaleIsSupported
- **GIVEN** window.moment is available and its locale() returns a code present in the supported locales
- **WHEN** locale is called
- **THEN** The detected locale code is returned (e.g. 'fr')

#### Scenario: FallsBackToNavigator.language
- **GIVEN** window.moment is unavailable and navigator.language returns a BCP-47 tag for a supported locale
- **WHEN** locale is called
- **THEN** The language subtag extracted from navigator.language is returned

#### Scenario: UnsupportedLocaleDefaultsToEnglish
- **GIVEN** The detected language code is not one of the supported locales (en, fr, es)
- **WHEN** locale is called
- **THEN** 'en' is returned as the default locale

### Requirement: QuickManualEntryStrings
The system SHALL add the following new translation keys to all three locale files (en.json, fr.json, es.json). Keys SHALL follow the existing dot-notation naming convention.

New keys required:

| Key | EN value |
|-----|----------|
| `modal.tab_from_file` | `From File` |
| `modal.tab_manual` | `Quick/Manual Entry` |
| `modal.manual_title_label` | `Title` |
| `modal.manual_title_placeholder` | `Untitled` |
| `modal.manual_date_label` | `Date` |
| `modal.manual_start_label` | `Start` |
| `modal.manual_end_label` | `End` |
| `modal.manual_create_button` | `Create` |
| `settings.time_increment.name` | `Time increment` |
| `settings.time_increment.desc` | `Default duration for manual meeting entries and time rounding` |

#### Scenario: AllKeysDefinedInAllLocales
- **WHEN** the plugin loads with locale set to fr or es
- **THEN** all new keys resolve to a non-empty string (no fallback to key name)

#### Scenario: ManualTabLabelLocalised
- **WHEN** the modal opens in French locale
- **THEN** the "Quick/Manual Entry" tab renders the French translation of `modal.tab_manual`

#### Scenario: UntitledPlaceholderLocalised
- **WHEN** the manual entry tab is open in Spanish locale
- **THEN** the title input placeholder text is the Spanish translation of `modal.manual_title_placeholder`

### Requirement: NewCommandAndRibbonStrings
The system SHALL add the following keys to all three locale files (en.json, fr.json, es.json):

| Key | EN value |
|-----|----------|
| `command.quick_name` | `New quick meeting note` |
| `settings.show_ribbon.name` | `Show ribbon icon` |
| `settings.show_ribbon.desc` | `Display the calendar icon in the ribbon bar. Takes effect after reloading Obsidian.` |

#### Scenario: NewKeysDefinedInEnglish
- **WHEN** the plugin loads with the EN locale
- **THEN** `command.quick_name`, `settings.show_ribbon.name`, and `settings.show_ribbon.desc` each resolve to a non-empty string

### Requirement: ExistingCommandKeyUpdated
The value of the existing key `command.name` SHALL be updated in all three locale files to reflect the renamed command.

| Key | Updated EN value |
|-----|-----------------|
| `command.name` | `Open meeting note importer` |

#### Scenario: AllNewKeysDefinedInAllLocales
- **WHEN** the plugin loads with locale set to fr or es
- **THEN** all new keys resolve to a non-empty string

#### Scenario: UpdatedCommandNameShownInHotkeys
- **WHEN** the user opens Obsidian Settings → Hotkeys
- **THEN** the existing command displays "Open meeting note importer" in the active locale
