# API Specification

> Generated by openlore v1.0.0 on 2026-06-18 09:21

## Purpose

This document specifies the HTTP API exposed by the system.

## Requirements

### Requirement: APIAuthentication

The API SHALL require authentication via: None (Obsidian plugin context), None, None (vault write access via Obsidian App)

#### Scenario: AuthenticatedRequest
- **GIVEN** a request with valid authentication credentials
- **WHEN** the request is processed
- **THEN** the request is authenticated successfully

#### Scenario: UnauthenticatedRequest
- **GIVEN** a request without authentication
- **WHEN** accessing a protected endpoint
- **THEN** the response status is 401 Unauthorized

### Requirement: COMMANDMeetingNote

The API SHALL support `COMMAND import-ics-meeting-note` to opens the ics drop modal for importing a meeting note from an .ics file or outlook drag payload.

#### Scenario: TriggerViaCommandPalette
- **GIVEN** The plugin is loaded and the user opens the Obsidian command palette
- **WHEN** The user selects the import meeting note command
- **THEN** The IcsDropModal opens with a drag-and-drop zone and a file picker button

#### Scenario: TriggerViaRibbonIcon
- **GIVEN** The plugin is loaded and the ribbon is visible
- **WHEN** The user clicks the calendar-plus ribbon icon
- **THEN** The IcsDropModal opens with a drag-and-drop zone and a file picker button

### Requirement: EVENTMeetingNote

The API SHALL support `EVENT IcsDropModal.onDrop(DragEvent)` to handles a drag-and-drop event on the modal drop zone. tries every possible transfer format (file, text/calendar, text/plain) to extract raw ical or outlook plain-text content, then parses and creates a meeting note.

**Request:**

```json
{
  "dataTransfer.files[0]": "File (.ics)",
  "dataTransfer.getData('text/calendar')": "string (raw iCal)",
  "dataTransfer.getData('text/plain')": "string (Outlook plain-text or raw iCal)"
}
```

**Response:**

```json
{
  "createdFile": "TFile",
  "opened": "boolean"
}
```

#### Scenario: Drop.icsFile
<!-- openlore-test: tags=smoke (auto) -->
- **GIVEN** User has an .ics file from their calendar
- **WHEN** User drops the file onto the modal drop zone
- **THEN** File is read as text, parsed as iCal, and a meeting note is created in the target folder
- **AND** If openAfterCreate is true the new note is opened in the editor
- **AND** Modal closes after successful creation

#### Scenario: DropOutlookMeetingInvite(macOS)
<!-- openlore-test: tags=smoke (auto) -->
- **GIVEN** User drags a meeting invite directly from Outlook for Mac
- **WHEN** User drops it onto the modal drop zone
- **THEN** text/calendar or text/plain payload is extracted, parsed, and a meeting note is created

#### Scenario: DropPlainTextOutlookSummary
<!-- openlore-test: tags=smoke (auto) -->
- **GIVEN** User copies and drags plain-text meeting summary from Outlook
- **WHEN** User drops it onto the modal drop zone
- **THEN** Outlook plain-text parser extracts meeting fields and creates a note

#### Scenario: DropUnsupportedFormat
<!-- openlore-test: tags=smoke,regression (auto) -->
- **GIVEN** User drops a non-calendar file or empty payload
- **WHEN** None of the transfer formats yield parseable content
- **THEN** An error notice is shown and no note is created

### Requirement: FUNCTIONMeetingNote

The API SHALL support `FUNCTION createMeetingNote(app: App, event: MeetingEvent, settings: IcalMeetingNotesSettings): Promise<TFile>` to creates a markdown note in the vault from a parsed meetingevent. uses a custom template if configured, otherwise falls back to a built-in template. injects attendees under the configured heading and meeting url/description under the notes heading. does not overwrite an existing file with the same name.

**Request:**

```json
{
  "app": "Obsidian App instance",
  "event": "MeetingEvent",
  "settings": "IcalMeetingNotesSettings"
}
```

**Response:**

```json
{
  "file": "TFile"
}
```

#### Scenario: CreateNoteWithDefaultTemplate
<!-- openlore-test: tags=smoke (auto) -->
- **GIVEN** No templateFile is set in settings
- **WHEN** createMeetingNote is called
- **THEN** A note is created using the built-in template with {{date}} and {{title}} replaced

#### Scenario: CreateNoteWithCustomTemplate
<!-- openlore-test: tags=smoke (auto) -->
- **GIVEN** settings.templateFile points to a valid vault file
- **WHEN** createMeetingNote is called
- **THEN** The custom template is loaded and placeholders are substituted before the file is written

#### Scenario: NoteAlreadyExists
- **GIVEN** A note with the same sanitized title and date already exists in the target folder
- **WHEN** createMeetingNote is called
- **THEN** The existing TFile is returned without overwriting

#### Scenario: SaveToActiveFolder
<!-- openlore-test: tags=smoke (auto) -->
- **GIVEN** settings.useActiveFolder is true and an active editor file is open
- **WHEN** createMeetingNote is called
- **THEN** Note is created in the same folder as the currently open file

#### Scenario: SaveToConfiguredFolder
<!-- openlore-test: tags=smoke (auto) -->
- **GIVEN** settings.useActiveFolder is false and settings.notesFolder is set
- **WHEN** createMeetingNote is called
- **THEN** Note is created in settings.notesFolder, creating intermediate directories as needed

### Requirement: FUNCTIONMeetingEvent

The API SHALL support `FUNCTION parseIcs(raw: string): MeetingEvent` to parses a raw ical (.ics) string using ical.js, extracts the vevent component, strips teams boilerplate from the description, and returns a structured meetingevent object.

**Request:**

```json
{
  "raw": "string"
}
```

**Response:**

```json
{
  "title": "string",
  "date": "string (YYYY-MM-DD)",
  "startTime": "string (HH:mm local)",
  "endTime": "string (HH:mm local)",
  "organizer": "string",
  "attendees": "string[]",
  "description": "string",
  "location": "string",
  "meetingUrl": "string"
}
```

#### Scenario: ParseStandardICalFile
- **GIVEN** A valid .ics file string containing a VEVENT component
- **WHEN** parseIcs is called with the raw string
- **THEN** Returns a MeetingEvent with all fields populated from the iCal properties

#### Scenario: ParseICalWithTeamsMeetingLink
- **GIVEN** iCal description contains a Microsoft Teams join URL
- **WHEN** parseIcs is called
- **THEN** meetingUrl is extracted from the description and Teams boilerplate text is stripped

#### Scenario: ParseEmptyOrInvalidICal
<!-- openlore-test: tags=regression (auto) -->
- **GIVEN** Raw string is empty or has no VEVENT
- **WHEN** parseIcs is called
- **THEN** Throws an Error

### Requirement: FUNCTIONMeetingEvent

The API SHALL support `FUNCTION parseOutlookText(text: string): MeetingEvent | null` to parses a plain-text meeting summary pasted or dragged from outlook for mac. returns null if the text does not match the expected outlook format (no 'when:' line).

**Request:**

```json
{
  "text": "string"
}
```

**Response:**

```json
{
  "title": "string",
  "date": "string (YYYY-MM-DD)",
  "startTime": "string (HH:mm local)",
  "endTime": "string (HH:mm local)",
  "organizer": "string",
  "attendees": "string[]",
  "description": "string",
  "location": "string",
  "meetingUrl": "string"
}
```

#### Scenario: ParseValidOutlookPlainTextSummary
<!-- openlore-test: tags=smoke (auto) -->
- **GIVEN** Text contains Subject, When:, and optionally Where:, Note:, Invitees: fields
- **WHEN** parseOutlookText is called
- **THEN** Returns a fully populated MeetingEvent

#### Scenario: ParseTextWithoutWhen:Line
- **GIVEN** Plain text that does not contain a 'When:' field
- **WHEN** parseOutlookText is called
- **THEN** Returns null

### Requirement: FUNCTIONMeetingEvent

The API SHALL support `FUNCTION stripTeamsBoilerplate(text: string): string` to removes microsoft teams meeting join-link boilerplate section from a meeting description. recognises boilerplate headers in english, french, and spanish.

**Request:**

```json
{
  "text": "string"
}
```

**Response:**

```json
{
  "cleaned": "string"
}
```

#### Scenario: StripEnglishTeamsBoilerplate
- **GIVEN** Description contains 'Microsoft Teams meeting' boilerplate in English
- **WHEN** stripTeamsBoilerplate is called
- **THEN** Returns description with everything from the boilerplate marker onward removed

#### Scenario: NoBoilerplatePresent
- **GIVEN** Description has no Teams boilerplate
- **WHEN** stripTeamsBoilerplate is called
- **THEN** Returns the original text unchanged

```json
  "templateFile": "string",
  "attendeesHeading": "string",
  "notesHeading": "string",
  "openAfterCreate": "boolean"
}
```

**Response:**

```json
{
  "notesFolder": "string",
  "useActiveFolder": "boolean",
  "templateFile": "string",
  "attendeesHeading": "string",
  "notesHeading": "string",
  "openAfterCreate": "boolean"
}
```

#### Scenario: LoadDefaultSettings
<!-- openlore-test: tags=smoke (auto) -->
- **GIVEN** Plugin is installed for the first time with no saved data
- **WHEN** loadSettings is called
- **THEN** Settings are initialised with defaults: useActiveFolder=true, openAfterCreate=true,
attendeesHeading='## Attendees', notesHeading='## Invite Notes', notesFolder='', templateFile=''

#### Scenario: PersistSettingsChange
- **GIVEN** User changes a setting in the plugin settings tab
- **WHEN** saveSettings is called
- **THEN** Updated settings are written to Obsidian's plugin data store and take effect on the next import

### Requirement: FUNCTIONSettings

The API SHALL support `FUNCTION resolveTargetFolder(app: App, settings: IcalMeetingNotesSettings): string` to determines the vault folder path where a new meeting note should be created, based on plugin settings and the currently active editor file.

**Request:**

```json
{
  "app": "Obsidian App instance",
  "settings": "IcalMeetingNotesSettings"
}
```

**Response:**

```json
{
  "folderPath": "string"
}
```

#### Scenario: UseActiveFileFolder
- **GIVEN** settings.useActiveFolder is true and an editor file is open
- **WHEN** resolveTargetFolder is called
- **THEN** Returns the parent folder path of the currently active file

#### Scenario: UseConfiguredFolder
- **GIVEN** settings.useActiveFolder is false
- **WHEN** resolveTargetFolder is called
- **THEN** Returns settings.notesFolder (empty string for vault root)

### Requirement: CONFIGIcalMeetingNotesSettings

The API SHALL support `CONFIG settings.useActiveFolder` to toggle whether new meeting notes are saved in the folder of the currently open file instead of the configured notesfolder.

**Request:**

```json
{
  "useActiveFolder": "boolean"
}
```

**Response:**

```json
{
  "saved": "boolean"
}
```

#### Scenario: EnableActiveFolderMode
<!-- openlore-test: tags=smoke (auto) -->
- **GIVEN** Plugin settings are open
- **WHEN** User toggles useActiveFolder to true
- **THEN** Setting is persisted and new notes are created in the folder of the active file

#### Scenario: DisableActiveFolderMode
- **GIVEN** Plugin settings are open
- **WHEN** User toggles useActiveFolder to false
- **THEN** Setting is persisted and new notes fall back to the configured notesFolder

### Requirement: CONFIGIcalMeetingNotesSettings

The API SHALL support `CONFIG settings.notesFolder` to set the vault folder path where newly created meeting notes are saved. empty value means vault root.

**Request:**

```json
{
  "notesFolder": "string"
}
```

**Response:**

```json
{
  "saved": "boolean"
}
```

#### Scenario: SetNotesFolderViaTextInput
- **GIVEN** Plugin settings are open
- **WHEN** User types a folder path in the notes folder field
- **THEN** Trimmed value is persisted as notesFolder

#### Scenario: SetNotesFolderViaAutocompleteSuggestion
- **GIVEN** Plugin settings are open and vault contains folders
- **WHEN** User selects a folder from the FolderSuggest dropdown

#### Scenario: ClearNotesFolder

- **THEN** notesFolder is saved as empty string, meaning vault root is used
