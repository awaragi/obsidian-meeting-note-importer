# Architecture Specification

> Generated by openlore v1.0.0 on 2026-06-18 09:21

## Purpose

This document describes the architectural patterns and structure of the system.

## Architecture Style

Monolith with domain-separated services. All code runs in a single Obsidian plugin process; there is
no client/server split or message bus. The four domains (settings, ical-parser, note-creator, i18n)
are organized as cohesive service classes rather than independent modules, which is appropriate
given the small surface area and the tight coupling required by the Obsidian plugin API lifecycle.

## Requirements

### Requirement: LayeredArchitecture

The system SHALL maintain separation between:
- Plugin Entry Point (Owns the Obsidian lifecycle (load/unload), registers commands and ribbon UI, and orchestrates the end-to-end import pipeline by wiring together the modal, parser, and note-creator.)
- UI / Modal (Presents the drag-and-drop import surface to the user. Handles both file-picker and drag-event input paths, then delegates raw content to the plugin's processing pipeline.)
- Parsing Domain (Normalizes two heterogeneous input formats (ICS and Outlook plain-text) into a single MeetingEvent structure. Also responsible for sanitizing content by stripping Microsoft Teams boilerplate.)
- Note Creation Domain (Resolves the destination folder, renders the Markdown note from a template (built-in or user-supplied), injects structured attendee and invite-notes blocks under configured headings, and writes the file to the vault.)
- Settings Domain (Persists and exposes user-configurable behaviour: target folder, template file path, heading names for content injection, active-folder toggle, and post-create open behaviour.)
- Internationalisation (i18n) (Detects the user's locale from Obsidian/browser context and returns translated strings for en, fr, and es with optional variable interpolation. Acts as a shared utility called across all other domains.)

#### Scenario: LayerSeparation
- **GIVEN** a request from the presentation layer
- **WHEN** business logic is needed
- **THEN** the presentation layer delegates to the business layer
- **AND** direct database access from presentation is prohibited

### Requirement: SecurityModel

The system SHALL implement security via: No authentication or authorization layer exists; the plugin operates entirely within the user's local Obsidian vault and never makes outbound network requests for core functionality. Trust is implicit — the plugin runs with the same filesystem permissions as the Obsidian process. The only external data boundary is the raw ICS/text content dragged in by the user; the parser sanitizes Teams boilerplate but performs no further input validation or sandboxing.

#### Scenario: AuthenticatedAccess
- **GIVEN** an unauthenticated request
- **WHEN** accessing protected resources
- **THEN** access is denied

## Layer Structure

### Plugin Entry Point

**Purpose**: Owns the Obsidian lifecycle (load/unload), registers commands and ribbon UI, and orchestrates the end-to-end import pipeline by wiring together the modal, parser, and note-creator.
**Location**: `src/ts/main.ts — IcalMeetingNotesPlugin, src/ts/main.ts — processEvent (async), src/ts/main.ts — processIcsContent (async)`

### UI / Modal

**Purpose**: Presents the drag-and-drop import surface to the user. Handles both file-picker and drag-event input paths, then delegates raw content to the plugin's processing pipeline.
**Location**: `src/ts/main.ts — IcsDropModal, src/ts/main.ts — handleDrop, src/ts/main.ts — onOpen, src/ts/main.ts — readFileAsText`

### Note Creation Domain

### Settings Domain

**Purpose**: Persists and exposes user-configurable behaviour: target folder, template file path, heading names for content injection, active-folder toggle, and post-create open behaviour.

**Location**: `src/ts/i18n.ts — I18nService, src/ts/i18n.ts — t (hub function, fanIn=9)`

## Data Flow

User triggers 'import-ics-meeting-note' command (or ribbon click) → IcalMeetingNotesPlugin opens
IcsDropModal → user drops an .ics file, drags an Outlook text payload, or picks a file via browser
dialog → IcsDropModal reads raw content (File API / DataTransfer API) and calls processIcsContent →
IcalParserService.parseIcs (ical.js) or IcalParserService.parseOutlookText produces a MeetingEvent
(Teams boilerplate stripped inline) → NoteCreatorService.createMeetingNote resolves target folder
via settings, renders template, injects attendee and notes blocks, writes TFile to Obsidian vault →
if settings.openAfterCreate is true, plugin opens the new note in the editor
